# description

- 类型：`string`
- 说明：文章的高密摘要（≤160 字符），用于列表预览与 SEO，提供读前价值过滤。

## 规则 (严格执行)
1. **字符红线**：绝对控制在 160 个字符以内（约 1-2 句话），符合标准 SEO 与卡片预览限制。
2. **全貌概括**：客观陈述文章的**核心主题**、**关键内容**与**阅读价值**。必须能让读者在 3 秒内判断内容相关性。
3. **禁止废话（红线）**：绝不可使用“本文主要”、“这篇文章将探讨”、“作者分享了”等无意义的铺垫语，直接以核心名词或动名词开篇。
4. **区别于 tldr**：`tldr` 只讲最终结论，而 `description` 必须包含内容框架的缩影。

## 示例 (Few-Shot)
- 长篇前端构建工具迁移教程与性能对比
  -> ✅ "深入解析前端工程从 Webpack 迁移至 Vite 的全流程，涵盖配置转化、插件替换策略及冷启动提速 10 倍的真实性能对比数据。"
  -> ❌ "本文将为您详细介绍如何把项目从 Webpack 迁移到 Vite，并分享过程中的踩坑记录，希望能帮助到大家。" (含大量铺垫废话，缺乏具体信息点)
- 一篇关于大模型 RAG 架构的深度技术笔记
  -> ✅ "全面剖析检索增强生成（RAG）技术的核心架构与痛点，对比主流向量数据库选型，并提供企业级知识库落地的最佳实践。"
  -> ❌ "RAG 技术能解决大模型的幻觉问题，非常值得在企业内推广部署。" (这是 tldr 的写法，缺乏对文章全貌的概括)
